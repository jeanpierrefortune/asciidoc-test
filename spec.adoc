As part of the *Reader Layer*, the *Reader API* provides high-level functionality for identifying, monitoring,
configuring readers and process card selection scenarios.

[[overview]]
=== Overview

The *Reader API* is a high-level interface used by terminal applications to operate smart card readers, it:

* provides the means to identify card readers and verify the presence of smart cards.
* offers the possibility to configure the observation of readers in order to be notified on card insertion or withdrawal.
* is the entry point to manage the selection of a smart card in a specific reader.

[[readerapifactory]]
=== ReaderApiFactory

[[readerapifactory-role]]
==== Role

`ReaderApiFactory` exposes the constructors for the public selection-related types of the
Reader API:

* xref:cardselectionmanager[`CardSelectionManager`] — the selection-scenario service;
* `BasicCardSelector` — a generic card selector (filters by protocol or power-on data;
see Phase 2.11 for the `CardSelector` family);
* `IsoCardSelector` — an ISO 7816-4 card selector.

Each factory method returns a *fresh instance*: distinct calls produce distinct objects.
The way the application obtains a `ReaderApiFactory` is implementation-defined (typically via
the implementation’s bootstrap service).

[[readerapifactory-createcardselectionmanager]]
=== ReaderApiFactory.createCardSelectionManager()

[[readerapifactory-createcardselectionmanager-intent]]
==== Intent

Returns a new instance of xref:cardselectionmanager[`CardSelectionManager`].

[[readerapifactory-createcardselectionmanager-postconditions]]
==== Post-conditions

* The returned value is *non-null* and implements `CardSelectionManager`.
* The returned manager is in its *initial state* — empty scenario, single-selection mode
(default), no release-channel request (default). See
xref:cardselectionmanager-concepts[`CardSelectionManager → Concepts`].
* Each call returns a *distinct* manager instance.

[[readerapiproperties]]
=== ReaderApiProperties

[[readerapiproperties-role]]
==== Role

Static container for the published properties of the Reader API. Currently exposes a single
field, xref:readerapiproperties-version[`VERSION`], identifying the spec minor version
implemented by the local Java binding.

[[readerapiproperties-version]]
=== ReaderApiProperties.VERSION

[[readerapiproperties-version-intent]]
==== Intent

Identifies the *minor version* of the Reader API specification that the local distribution
of the Java binding implements (e.g. `"2.1"`).

[[readerapiproperties-version-postconditions]]
==== Post-conditions

* The value is a *non-empty* text string in `++<++major++>++.++<++minor++>++` form.
* The value is a *compile-time constant* — it does not vary at runtime.

[[readerapiproperties-version-behavior]]
==== Behavior

* The third (patch) digit is *deliberately omitted*: a `++<++major++>++.++<++minor++>++` value covers any
patch release within that minor — see the
link:specifications/overview#versioning-and-lifecycle[Versioning & lifecycle]
note on the Specifications overview.
* The value matches the minor-version segment of the spec URL that documents the contract this
distribution implements: for `VERSION = "2.1"`, the spec page is
`/specifications/reader-layer/reader-api/2.1/`.

[[readerapiproperties-version-conformance]]
==== Conformance requirements

[width="100%",cols="34%,33%,33%",options="header",]
|===
|ID |Statement |Verifiable as
|*CONF-R-PROPS-VERSION-01* |`VERSION` is a non-empty `++<++major++>++.++<++minor++>++` string equal to the spec page minor version the distribution implements. |Read the constant; assert it is non-empty and matches `++\++d{plus}++\++.++\++d{plus}`. If the distribution claims conformance to spec page `/X.Y/`, assert `VERSION` equals `X.Y`.
|===

[[cardreader]]
=== CardReader

[[cardreader-role]]
==== Role

Any smart card reader entity exposed by an implementation of this API is accessible through an
instance of `CardReader`. The interface exposes:

* a stable xref:cardreader-getname[name] identifying the reader;
* the reader’s xref:cardreader-iscontactless[communication mode], contactless or contact;
* the xref:cardreader-iscardpresent[current card-presence state].

A reader may be a contactless or a contact reader. Optional capabilities (observation, protocol
configuration) are described by the xref:observablecardreader[`ObservableCardReader`] and
xref:configurablecardreader[`ConfigurableCardReader`] interfaces.

[[observablecardreader]]
=== ObservableCardReader

[[observablecardreader-role]]
==== Role

A reader implementing `ObservableCardReader` dynamically detects card insertion and removal and
notifies registered observers via xref:cardreaderevent[card reader events]. The interface extends
xref:cardreader[`CardReader`] with:

* the xref:observablecardreader-setreaderobservationexceptionhandler[exception-handler registration]
(mandatory before activating observation);
* observer registration and lifecycle:
xref:observablecardreader-addobserver[`addObserver`],
xref:observablecardreader-removeobserver[`removeObserver`],
xref:observablecardreader-clearobservers[`clearObservers`],
xref:observablecardreader-countobservers[`countObservers`];
* the detection lifecycle:
xref:observablecardreader-startcarddetection[`startCardDetection`],
xref:observablecardreader-stopcarddetection[`stopCardDetection`],
xref:observablecardreader-finalizecardprocessing[`finalizeCardProcessing`];
* two enum types describing detection behaviour:
xref:observablecardreader-detectionmode[`DetectionMode`],
xref:observablecardreader-notificationmode[`NotificationMode`].

[[observablecardreader-states]]
==== State transitions

The state machine of an observable reader (insertion / processing / removal) is described in the
dedicated xref:observable-reader-states[Observable reader states] section of this page. The
individual operation sections that follow refer to that state machine when relevant.

[[observablecardreader-setreaderobservationexceptionhandler]]
=== ObservableCardReader.setReaderObservationExceptionHandler()

[[observablecardreader-setreaderobservationexceptionhandler-intent]]
==== Intent

Registers the *exception handler* that will receive notifications of fatal errors occurring
during reader observation.

[[observablecardreader-setreaderobservationexceptionhandler-preconditions]]
==== Pre-conditions

* The provided handler is non-null.

[[observablecardreader-setreaderobservationexceptionhandler-behavior]]
==== Behavior

* Invocation of this method is *mandatory before* activating observation through
xref:observablecardreader-startcarddetection[`startCardDetection`].
* The handler is *replaceable*: invoking this method again with a non-null handler replaces
the previously registered handler. The replacement takes effect for any error notification
delivered *after* the call returns. Notifications already being processed by the previous
handler are not retargeted.

[[observablecardreader-addobserver]]
=== ObservableCardReader.addObserver()

[[observablecardreader-addobserver-intent]]
==== Intent

Registers an observer that will receive xref:cardreaderevent[card reader events] produced by this
reader (insertion, matched, removal, …).

[[observablecardreader-addobserver-preconditions]]
==== Pre-conditions

* The provided observer is non-null.

[[observablecardreader-addobserver-behavior]]
==== Behavior

* After a successful call, the observer receives every event delivered by the reader, alongside
the other registered observers.

no-op*: no exception is raised, no duplicate notification path is created, and the value
returned by xref:observablecardreader-countobservers[`countObservers`] is unchanged.

[[observablecardreader-startcarddetection-postconditions]]

[[observablecardreader-finalizecardprocessing]]
=== ObservableCardReader.finalizeCardProcessing()

[[observablecardreader-finalizecardprocessing-intent]]
==== Intent

Signals to the observable reader that the application has finished processing the card. After
this call returns, the reader is free to advance its state machine to the next stage (typically
waiting for the card to be removed; see xref:observable-reader-states[Observable reader states]).

[[observablecardreader-finalizecardprocessing-preconditions]]
==== Pre-conditions

None. The call is *safe in any state* of the observable reader: it may be invoked before
card detection has started, after it has been stopped, or in any of the observation
sub-states. Calls that do not match the _"`Wait for card processing`"_ state are silently
absorbed (no exception, no state change). See CONF-R-OBS-FINAL-03 below.

[[observablecardreader-finalizecardprocessing-postconditions]]
==== Post-conditions

* If the physical communication channel between the reader and the card was open at call time,
it is *closed* when the call returns.
* If the physical channel was already closed at call time, the call has *no observable effect*
on it (idempotence).
* The observable reader’s state machine is free to leave the _"`Wait for card processing`"_
state. The actual transition is governed by the state machine described in
xref:observable-reader-states[Observable reader states] and is not altered by this method
itself.

[[observablecardreader-finalizecardprocessing-behavior]]
==== Behavior

* The method is *idempotent*: repeated invocations with the channel already closed are no-ops.
* The method is the *recovery path* when the normal channel-close (which the reader carries
out as part of the last APDU exchange with the card) did not happen — for example because the
card-processing code was interrupted by an error.
* The method is a *silent no-op* when the reader is in a state where the _"`Wait for card
processing`"_ → next-state transition does not apply (e.g. before `startCardDetection`, after
`stopCardDetection`, during _"`Wait for card insertion`"_, or during _"`Wait for card
removal`"_). This guarantees that the recommended usage from a "`finally`"-style construct (see
Notes) is always safe.

[[observablecardreader-finalizecardprocessing-errors]]
==== Error cases

The call does not signal errors of its own: it succeeds whether or not the channel was open at
entry, and whether or not the reader is in the _"`Wait for card processing`"_ state.

[[observablecardreader-finalizecardprocessing-conformance]]
==== Conformance requirements

[[configurablecardreader]]
=== ConfigurableCardReader

[[configurablecardreader-role]]
==== Role

A reader implementing `ConfigurableCardReader` exposes operations to manage the *card
communication protocols* it uses. The interface extends xref:cardreader[`CardReader`] with:

* xref:configurablecardreader-activateprotocol[activation of a protocol], binding a physical
protocol (as known by the reader) to a logical protocol name (as used by the application
during card selection);
* xref:configurablecardreader-deactivateprotocol[deactivation of a protocol];
* xref:configurablecardreader-getcurrentprotocol[query of the physical protocol of the most-recently selected card].

[[configurablecardreader-concepts]]
==== Concepts

* *Physical protocol name* — the name by which the reader identifies a communication
technology over the air (e.g. ISO 14443-A, ISO 14443-B, etc.). The set of supported physical
protocol names is documented by the reader.
* *Logical protocol name* — an application-chosen name attached to a physical protocol. The
application uses the logical name when filtering during card selection (see the
`CardSelector.filterByCardProtocol` operation on `CardSelector`). This mapping lets the
application use a single logical name across different reader implementations that may
identify the same physical technology under different names — particularly useful for
non-ISO cards.

[[configurablecardreader-activateprotocol-intent]]
==== Intent

[[configurablecardreader-activateprotocol-preconditions]]
==== Pre-conditions

* Both `physicalProtocolName` and `logicalProtocolName` are non-null and non-empty.
* `physicalProtocolName` matches one of the protocols supported by the reader (consult the

[[configurablecardreader-activateprotocol-postconditions]]

* An association is registered between `physicalProtocolName` and `logicalProtocolName`. The

[[cardselectionmanager]]
=== CardSelectionManager

[[cardselectionmanager-role]]
==== Role

`CardSelectionManager` is the service that builds and executes a *card selection scenario*.
The scenario is an ordered sequence of selection cases, each targeting a specific card
definition provided by a `CardSelectionExtension` (see the `selection.spi` package). The
manager runs the scenario synchronously against a card already present on a reader, or
schedules it to run automatically when a card is presented to an
xref:observablecardreader[`ObservableCardReader`].

An instance is obtained via `ReaderApiFactory.createCardSelectionManager` (see
`ReaderApiFactory` in the `reader` package).

[[cardselectionmanager-concepts]]
==== Concepts

* *Selection case* — one target card definition: a `CardSelector` (filtering rules) plus the
`CardSelectionExtension` that interprets the response. A case may also carry follow-up
commands to be issued after a successful selection.
* *Scenario* — the ordered sequence of selection cases prepared on the manager. The order
is the order in which cases were appended via
xref:cardselectionmanager-prepareselection[`prepareSelection`] (or imported via
xref:cardselectionmanager-importcardselectionscenario[`importCardSelectionScenario`]).
* *Single-selection mode (default)* — when the manager runs the scenario, it stops as soon
as one case succeeds. Subsequent cases are not executed.
* *Multiple-selection mode* — enabled via
xref:cardselectionmanager-setmultipleselectionmode[`setMultipleSelectionMode`]. When the
manager runs the scenario, *every* case is executed, regardless of earlier successes.
* *Channel after selection* — by default the physical channel remains open after the
scenario completes (so the application can run a transaction). To request the channel be
closed at the end of the scenario, call
xref:cardselectionmanager-preparereleasechannel[`prepareReleaseChannel`] before executing.
* *Synchronous vs. scheduled execution* — see xref:card-selection-modes[Card selection modes].

[[cardselectionmanager-lifecycle]]
==== Lifecycle of a scenario

[arabic]
. *Prepare* the scenario by appending one or more cases with
xref:cardselectionmanager-prepareselection[`prepareSelection`]. Optionally enable
xref:cardselectionmanager-setmultipleselectionmode[multiple-selection mode] and request
xref:cardselectionmanager-preparereleasechannel[channel release].
. *Execute* the scenario, either synchronously on a present card
(xref:cardselectionmanager-processcardselectionscenario[`processCardSelectionScenario`]),
or asynchronously by scheduling on an observable reader
(xref:cardselectionmanager-schedulecardselectionscenario[`scheduleCardSelectionScenario`]).
. For scheduled execution, the result is delivered via a `CardReaderEvent` and analyzed
with
xref:cardselectionmanager-parsescheduledcardselectionsresponse[`parseScheduledCardSelectionsResponse`].
. Optionally *export* the processed scenario for analysis in a different environment via
xref:cardselectionmanager-exportprocessedcardselectionscenario[`exportProcessedCardSelectionScenario`]
/ xref:cardselectionmanager-importprocessedcardselectionscenario[`importProcessedCardSelectionScenario`].

[[cardselectionmanager-setmultipleselectionmode]]
=== CardSelectionManager.setMultipleSelectionMode()

[[cardselectionmanager-setmultipleselectionmode-intent]]
==== Intent

Enables *multiple-selection mode* on this manager: when running the scenario, every
selection case is processed, even when an earlier case has already succeeded.

[[cardselectionmanager-setmultipleselectionmode-postconditions]]
==== Post-conditions

* Subsequent calls to
xref:cardselectionmanager-processcardselectionscenario[`processCardSelectionScenario`] and
xref:cardselectionmanager-schedulecardselectionscenario[`scheduleCardSelectionScenario`]
execute every prepared selection case, regardless of earlier successes.

[[cardselectionmanager-setmultipleselectionmode-behavior]]

[[cardselector]]
=== CardSelector

[[cardselector-role]]
==== Role

`CardSelector` is the base contract of the family of objects that the application passes to
xref:cardselectionmanager-prepareselection[`CardSelectionManager.prepareSelection`] to describe
how a card should be matched. It carries a set of optional *filters* that restrict the
selection process.

Concrete selectors in the family:
- xref:basiccardselector[`BasicCardSelector`] — common filters only (no ISO-specific filter).
- xref:isocardselector[`IsoCardSelector`] — adds ISO 7816-4 selection filters via
xref:commonisocardselector[`CommonIsoCardSelector`].

Selectors are obtained via the factory methods of
xref:readerapifactory[`ReaderApiFactory`].

[[cardselector-rules]]
==== Filter combination rules

* Each filter on a selector is *optional*.
* Filters *may be combined*: multiple filters on the same selector all apply.
* When *no filter* is configured, any card that responds upon insertion is considered
matched by the selector.
* When *one or more* filters are configured, the card is matched only if *every* filter
accepts it; if any filter rejects, the card is not matched by the selector.

[[cardselector-conformance]]
==== Conformance requirements

[width="100%",cols="34%,33%,33%",options="header",]
|===
|ID |Statement |Verifiable as
|*CONF-R-SLCT-RULES-01* |A selector with no filter matches any card that responds upon insertion. |Build a selector and configure no filter; present an arbitrary card; assert the card is matched.
|*CONF-R-SLCT-RULES-02* |A selector with one or more filters matches a card only when every configured filter accepts it. |Configure two filters known to accept the same card; assert match. Configure two filters where one rejects the card; assert no match.
|===

* The two forms are interchangeable when they represent the same byte sequence.

[[commonisocardselector-filterbydfname-postconditions]]
==== Post-conditions

* The selector accepts a card only when its DF name starts with the provided AID.

[[cardselectionextension]]
=== CardSelectionExtension

[[cardselectionextension-role]]
==== Role

Service Provider Interface (SPI) implemented by *card extensions* (not by application
code). Used as the second argument of
xref:cardselectionmanager-prepareselection[`CardSelectionManager.prepareSelection`]. The
extension carries two responsibilities:

* *Enrich* the selection case with additional commands to be sent to the card after a

[[cardtransactionmanager]]
=== CardTransactionManager

[[cardtransactionmanager-role]]
==== Role

`CardTransactionManager` is the base SPI for executing *card transactions* — sequences of
APDU commands prepared by the application and then dispatched to the card in a single
processing call. The manager:

* holds a xref:smartcard[`SmartCard`] instance representing the card under transaction (the
"`registered card`");
* accumulates prepared commands;
* when asked to xref:cardtransactionmanager-processcommands[process the commands], sends the
APDUs to the card, retrieves the responses, updates the registered `SmartCard`, and
applies the requested xref:channelcontrol[`ChannelControl`] policy on the physical channel
at the end of the cycle.

Card-specific extensions (e.g. Calypso, Storage Card) define their own command-preparation
APIs by extending this base interface; this specification only standardises the common
execution contract.

[[cardtransactionmanager-model]]
==== Model

* *Prepare then process.* Command preparation is normatively a separate step from
execution. Multiple commands may be prepared before a single processing call.
* *Batching motivation.* Grouping commands minimises network exchanges between the
application and the card — particularly valuable in *distributed architectures* where
the manager and the reader sit on different network hops.
* *Registered SmartCard kept in sync.* The registered `SmartCard` instance is updated as
the transaction proceeds, so that the in-memory image stays consistent with the card’s
state after each successful exchange.

[[cardtransactionmanager-processcommands]]
=== CardTransactionManager.processCommands()

[[cardtransactionmanager-processcommands-intent]]
==== Intent

Processes all previously prepared commands as a single dispatch to the card and applies the
requested xref:channelcontrol[`ChannelControl`] policy on the physical channel at the end of
the cycle.

[[cardtransactionmanager-processcommands-preconditions]]
==== Pre-conditions

* `channelControl` is non-null.
* A `SmartCard` instance has been registered with the manager.
* The reader and card are reachable; the physical channel is open (or in a state from which
the transaction can be issued).

[[cardtransactionmanager-processcommands-postconditions]]
==== Post-conditions

* The APDUs of every prepared command have been sent to the card, *up to the first
failure* (see Behavior below), and their responses retrieved.
* The registered `SmartCard` has been updated to reflect the cumulative effect of the
successful responses.
* For *write commands*: the `SmartCard` update is performed *only when the command
succeeds*. A failed write leaves the previously held image untouched for that command.
* The physical channel is managed per the
xref:channelcontrol[`ChannelControl`] argument.

[[cardtransactionmanager-processcommands-behavior]]
==== Behavior

* *First-failure interruption.* When a command fails (transport error, unexpected status
word, or invalid response), the manager stops the cycle immediately. Subsequent prepared
commands are not sent.

[[cardtransactionmanager-processcommands-errors]]
==== Error cases

[width="100%",cols="50%,50%",options="header",]
|===
|Condition (observable) |Outcome (language-agnostic)
|`channelControl` is null. |The call rejects the invalid argument (language-specific). The transaction state is unchanged.
|Communication with the reader fails during dispatch. |The call surfaces a *reader-communication failure*.
|Communication with the card fails (transport error, unexpected status word). |The call surfaces a *card-communication failure*.
|A command returns data that cannot be interpreted. |The call surfaces an *invalid-card-response failure*.
|===

[[cardtransactionmanager-processcommands-conformance]]
==== Conformance requirements

[width="100%",cols="34%,33%,33%",options="header",]
|===
|ID |Statement |Verifiable as
|*CONF-R-TXN-PROC-01* |After a successful `processCommands`, every prepared APDU has been sent to the card and the registered `SmartCard` reflects the cumulative effect of the successful responses. |Prepare N commands known to succeed against a known card; call the method; assert the `SmartCard` image matches the expected post-state.
|*CONF-R-TXN-PROC-02* |When any prepared command fails, the manager stops dispatching subsequent commands. xref:cardselectionmanager-exportprocessedcardselectionscenario[processed-scenario pair] —
exchange data as *stable, versioned text strings*.

[[selection-scenario-export-format-stability]]
==== Stability promise

* Each exported string *carries a format version field*.
* An implementation conformant with this specification at minor version `X.Y` must accept any
string produced by an implementation conformant with this specification at minor version
`X.Y` or any earlier minor version sharing the same major (i.e. forward compatibility is
guaranteed within a major).
* A format upgrade across major versions is allowed and may introduce non-backward-compatible
changes; such upgrades MUST be accompanied by a documented migration path.

[[selection-scenario-export-format-roundtrip]]
==== Round-trip property

* *Scenario format*: a scenario `S` exported by manager `M1` and imported into manager `M2`
produces a scenario behaviourally equivalent to `S` when executed (CONF-R-SEL-EXPSCEN-01 /
CONF-R-SEL-IMPSCEN-01).
* *Processed-scenario format*: a processed scenario exported by manager `M1` and imported
into manager `M2` (which has been configured with the same scenario as `M1` and has the
required card extensions installed) produces a `CardSelectionResult` equivalent to the
result that `M1` had (CONF-R-SEL-EXPPROC-01 / CONF-R-SEL-IMPPROC-01).

[[selection-scenario-export-format-schema]]
==== Schema definition

The concrete textual schema of the two formats is documented separately by the reference
implementation (Eclipse Keypop). The schema may evolve in a forward-compatible fashion
(per the Stability promise above). Implementers building a from-scratch conformant manager
MUST adhere to the same schema; their distribution carries the reference to the schema
documentation.

[[exception-categories]]
=== Exception categories

The Reader API surfaces failures through four normative *exception categories*. Each
category has a documented meaning, an "`information carried`" model, and a list of methods
that surface it. The conditions under which a category is raised are described in the
*Error cases* sub-section of each method that surfaces it, and tested by the corresponding
conformance requirements; there are intentionally *no per-category CONF requirements* at
this level.

The language-agnostic content below uses these terms:

* *Diagnostic message* — a free-form textual description of the failure context. Carried
by the exception instance.
* *Chained cause* — an inner exception/error value identifying the underlying failure
that triggered the condition, when one is available.

Each binding (Java, C{plus}{plus}, …) maps a category to one or more concrete exception types of that
language. The mapping for each supported binding is listed at the end of each sub-section.

[[exception-readercommunicationfailure]]
==== Reader-communication failure

Signals that the call could not complete because communication with the reader did not
succeed. The most likely cause is a physical disconnection of the reader, but other
transport-layer problems may also be the origin.

Information carried:
- A diagnostic message.
- Optionally, a chained cause.

Surfaced by:
- xref:cardreader-iscardpresent[`CardReader.isCardPresent`] — under CONF-R-RDR-PRESENT-02.
- xref:cardselectionmanager-processcardselectionscenario[`CardSelectionManager.processCardSelectionScenario`]
— under CONF-R-SEL-PROC-01.

Bindings:
- *Java*: `org.eclipse.keypop.reader.ReaderCommunicationException`.

[[exception-readerprotocolnotsupportedfailure]]
==== Reader-protocol-not-supported failure

Signals that the requested card-communication protocol is not supported by the reader.

Information carried:
- A diagnostic message identifying the offending physical protocol name.
- *No* chained cause for this category.

Surfaced by:
- xref:configurablecardreader-activateprotocol[`ConfigurableCardReader.activateProtocol`] —
under CONF-R-CFG-ACTPROTO-03.
- xref:configurablecardreader-deactivateprotocol[`ConfigurableCardReader.deactivateProtocol`]
— under CONF-R-CFG-DEACTPROTO-03.

Bindings:
- *Java*: `org.eclipse.keypop.reader.ReaderProtocolNotSupportedException`.

[[exception-cardcommunicationfailure]]
==== Card-communication failure

Signals that the call could not complete because communication with the card did not
succeed — transport error during APDU exchange, or unexpected status word when status-word
checking is enabled in the card request.

Information carried:
- A diagnostic message.
- Optionally, a chained cause.

Surfaced by:
- xref:cardselectionmanager-processcardselectionscenario[`CardSelectionManager.processCardSelectionScenario`]
— under CONF-R-SEL-PROC-01.

Bindings:
- *Java*: `org.eclipse.keypop.reader.CardCommunicationException`.

[[exception-invalidcardresponsefailure]]
==== Invalid-card-response failure

Signals that a response received from the card was invalid — structurally malformed,
semantically inconsistent, or otherwise uninterpretable.

Information carried:
- A diagnostic message.
- Optionally, a chained cause.

Surfaced by:
- xref:cardselectionmanager-processcardselectionscenario[`CardSelectionManager.processCardSelectionScenario`]
— under CONF-R-SEL-PROC-01.
- xref:cardselectionmanager-parsescheduledcardselectionsresponse[`CardSelectionManager.parseScheduledCardSelectionsResponse`]
— under CONF-R-SEL-PARSE-01.
- xref:cardselectionmanager-importprocessedcardselectionscenario[`CardSelectionManager.importProcessedCardSelectionScenario`]
— under CONF-R-SEL-IMPPROC-02.
- `CardTransactionManager.processCommands` in the `transaction.spi` package (Phase 2.14).

Bindings:
- *Java*: this category is currently split across *two coexisting classes* per the Q4
transitional decision —
- `org.eclipse.keypop.reader.InvalidCardResponseException` (`@since 2.1.0`) at the
`CardTransactionManager.processCommands` site;
- `org.eclipse.keypop.reader.selection.InvalidCardResponseException` (`@since 1.0.0`) at
the three selection-manager sites listed above.

The spec treats them as one logical category; the two-class split will be consolidated in
a future release.

[[card-selection-modes]]
=== Card selection modes

Depending on the card transaction use case, or on the reader capability, there are two ways to manage the selection of a
card:

* Either synchronously on a simple reader, a selection could be operated directly by transmitting the card selection
scenario. In this case the same entity manages both the card selection and the card processing.
* Otherwise asynchronously on an observable reader, a scheduled card selection could be defined.
In this case the card selection is operated automatically at the insertion of the card. In this case,
the card selection is next managed by the observable reader, but the card processing is managed by a reader observer.

[[observable-reader-states]]
=== Observable reader states

An observable reader is active only when at least one reader observer is registered, and if the start of the detection
has been requested.
When active, an observable reader could switch between three internal states:
++*++ Wait for card insertion
++*++ Wait for card processing
++*++ Wait for card removal

In the nominal case, a reader observer indicates to the observable reader that the processing of the card is finished by
releasing the card channel.
To manage a failure of the reader observer process, the observable reader interface provides also a method to finalize
the card processing.

.Reader API - Observable Reader States - Activity Diagram
image::/media/specifications/reader++_++management++_++reader++_++observation++_++state++_++diagram.svg[]

The states could be switched:
- due to an explicit API request (blue arrows):
- the release of the card channel,
- the call of an observable reader method:
- the addition or the remove of an observable reader,
- a request to start or stop the detection, to finalize the card processing.
- Or because of an external event (red arrows), the insertion or the remove of a card.
- the insertion of a card causing the observable reader to notify a `CARD++_++MATCHED` reader event (in case of successful
scheduled selection) or a `CARD++_++INSERTED` reader event (notification mode defined as `ALWAYS`).
- the removal of a card causing the observable reader to notify a `CARD++_++REMOVED` reader event.

If a card detection is started with the `REPEATING` detection mode, then later when the card is removed, the reader
starts again the detection of a new card.

Notification of card removal during the card removal wait state is a feature of any observable reader.
However, some observable readers may additionally have the ability to notify removal during the card processing state.

[[limitations-and-constraints]]
=== Limitations & constraints

On a specific card reader, the *Reader API* allows to address only a *single logical channel*.
